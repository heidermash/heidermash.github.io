01 · Architecture

One traceable path from source to decision.

Each layer has a bounded responsibility. Raw records remain auditable, invalid and superseded rows receive reason codes, business logic lives in dbt, and Power BI consumes curated marts rather than calculating around source defects.

STEP 01

Generate

Python creates versioned, source-shaped synthetic entities plus ground truth for injected issues.

→
STEP 02

Load

An idempotent batch loader reconciles physical files into local PostgreSQL raw tables.

→
STEP 03

Transform

dbt staging, intermediate, and mart layers type, quarantine, reconcile, and model the data.

→
STEP 04

Orchestrate

Airflow schedules generation, verification, loading, testing, export, and evidence gates.

→
STEP 05

Analyze

A generated PBIP connects Power BI Desktop to curated PostgreSQL marts in Import mode.

Reproducibility control: the same manifest is skipped on retry rather than appended again, while dbt still rebuilds and tests the analytical layers.

03 · Power BI Desktop

Four views, one governed semantic model.

The report is generated as source-controlled PBIP, PBIR, and TMDL files. It is designed for local Power BI Desktop authoring and refresh; it is not presented as a public Power BI Service deployment.

05 · Honest boundaries

What this project does—and does not—claim.

Portfolio engineering environment

  • Every customer, order, product, store, driver, promotion, and delivery record is deterministic and synthetic.
  • The stack runs locally with PostgreSQL, dbt Core, Apache Airflow, Docker Desktop, and Power BI Desktop.
  • No private research, employer, service-user, or personally identifiable data is included or published.
  • The project demonstrates comparable workflow skills; it does not claim production, cloud, SAP, Oracle, SQL Server, or proprietary consumer-intelligence experience.
  • Synthetic KPI values are engineering controls, not observed business performance, adoption, savings, or revenue impact.
